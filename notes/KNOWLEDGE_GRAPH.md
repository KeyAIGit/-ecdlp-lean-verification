# The ECDLP verified knowledge graph

`data/knowledge_graph.json` is a machine-readable, navigable index of everything
this repository has **machine-checked** about the ECDLP landscape for secp256k1.
It is generated by `scripts/build_knowledge_graph.py` from the human-maintained
sources of truth (it never asserts a proof — the Lean kernel remains the only
judge of correctness).

## Why it exists

The north star of this project is not to break secp256k1 (the generic lower bound
`Ω(√n)` makes that infeasible). It is to build a **verified, navigable substrate**:
a body of kernel-checked facts, their dependencies, and a precise map of the
foundations still missing — so that a future automated reasoner can load the
*known-and-verified* frontier of the problem directly, rather than re-deriving it
from general training data. The JSON graph is that substrate in loadable form.

## What's in it

| Section | Content |
|---|---|
| `theorems` | every proved theorem in `VERIFIED.md`: name, module, file, the corpus claim it verifies, proof method, and whether it is gated by the no-`sorry` CI build |
| `barriers` | the missing-foundation nodes from `BARRIERS.md` as first-class data: what is missing, what it blocks, the Mathlib area, and any partial progress |
| `corpus` | summary of the 486-claim corpus by `formal_status` (the candidate frontier) |
| `edges` | `imports` (file-level dependency between project modules) and `frontier_of` (a verified theorem sitting at the boundary of a barrier) |

Each theorem node also carries `area` (a coarse research facet: `curve-torsion`,
`generic-hardness`, `protocol-soundness`, `reduction`, `attack-resistance`,
`primality`, `params`) and `dependency_depth` (longest intra-project import chain
beneath its module). `counts.by_area` / `counts.by_method` summarise the corpus.

A human/AI-readable rendering is generated alongside the JSON at
`data/knowledge_graph.md` — the verified frontier grouped by area, then each
barrier with its verified frontier nodes. Edit the generator, not these outputs.

## Regenerate / check

```sh
python3 scripts/build_knowledge_graph.py          # rewrite data/knowledge_graph.json
python3 scripts/build_knowledge_graph.py --check   # fail if the committed file is stale
```

The generator reads `VERIFIED.md`, the `Ecdlp/**/*.lean` import lines,
`BARRIERS.md` (mirrored as curated barrier nodes), and the corpus CSV. After
promoting a new theorem (adding its row to `VERIFIED.md`), re-run the generator so
the graph stays in sync with the ledger.

## Honesty of the artifact

The graph is **derived**, not authoritative: it restates what the ledger and the
Lean sources already contain. A node appearing here is not evidence of a proof —
the evidence is the green `lake build` with the no-`sorry` gate. If a row is in
`knowledge_graph.json` but not actually built and gated, that is a generator bug,
not a verified fact. The `gated` flag records, per theorem, whether its module is
on the built import surface (`Ecdlp.lean`).
